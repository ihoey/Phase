你是资深 macOS 产品设计师 + SwiftUI 工程师，请帮我开发一款名为 Phase 的 macOS 原生代理工具。

产品定位：

- 功能上只做代理（不做抓包/HTTPS 解密）
- 体验上要明显优于 Surge / Loon / Quantumult X
- 重点是 UI 精致、信息层级清晰、交互优雅
- macOS 原生体验优先（SwiftUI，低功耗）

请输出：

1. UI 设计规范与布局建议（颜色、字体层级、间距、阴影、磨砂玻璃）
2. 关键页面结构设计：总览 / 规则 / 节点 / 日志 / 设置
3. SwiftUI 代码示例：卡片样式、状态标签、侧边栏 + 内容区
4. 交互细节建议：动画、状态切换、loading、错误提示

风格参考：

- iOS/macOS 原生感
- 简洁高级（类似 Raycast / Arc / Linear）
- 不要“工程感”，要“产品感”

【项目初心（最高优先级）】
我已经使用并购买了 Surge、Loon、Quantumult X。
我开发 Phase 的唯一根本动机是：

👉 我相信我可以把 UI 做得比它们都好。

如果 Phase 的 UI 不能在审美、克制感、系统融合度上明显优于这些产品，
那么这个项目在产品层面是失败的。

【产品定位】
Phase 是一个：

- 系统级
- 后台常驻
- 高性能、低功耗
- 面向“懂行用户”的专业网络工具

Phase 不是玩具，也不是展示技术的 Demo，
而是一个“每天开着、但几乎感觉不到它存在”的系统工具。

【UI 设计的最高原则（不可违背）】

1. UI 优先级 > 功能优先级

   - 功能可以慢慢加
   - UI 一旦定型就很难推翻
   - 任何功能设计，必须先回答“它是否破坏整体 UI 克制感”

2. 克制 > 炫技

   - 不做赛博朋克
   - 不做花哨渐变
   - 不做“网络地球 / 流量瀑布”
   - 不堆信息密度来显得“专业”

3. Phase 的 UI 应该让人联想到：
   - macOS 系统设置（新版）
   - Activity Monitor（更现代）
   - Raycast
   - Xcode 的 Source Control / Organizer
   - Arc
   - Linear

关键词是：
👉 System-native · Calm · Confident · Precise

【UI 体验的硬指标】

- 打开 Phase 后，用户在 0.5 秒内能明确知道：
  - 当前是否启用代理
  - 当前使用的节点 / 模式
- 不需要阅读、不需要理解术语
- 状态通过视觉层级自然呈现

【信息架构规则（非常重要）】

- 每一个页面，只回答一个问题
- 不允许“全都放在一个页面”
- 高级功能必须隐藏在二级或三级入口
- UI 永远不主动炫耀“我支持多少协议”

【技术选型（不可偏离）】

- 平台：macOS
- UI：SwiftUI 为主，必要时使用 AppKit
- 禁止使用 Web UI、Electron、Flutter
- 禁止为了“快”牺牲系统一致性
- 底层代理使用 sing-box，UI 不参与任何网络实现
- 以最佳实践组织

【代码与 UI 的关系】

- View 层只负责展示
- ViewModel 层负责“如何呈现状态”
- 任何业务逻辑如果污染 View，都必须被拆出去
- 所有 SwiftUI 代码必须考虑真实产品可维护性
- 主动指出“这个设计是否过于工程化 / 不够优雅”
- UI 动画：
  - 短
  - 克制
  - 有目的
  - 绝不为了“好看”而存在

你接下来提供的所有建议、设计和代码，
都必须把“Phase 的 UI 是否足够好”作为第一判断标准。


- 订阅：兼容主流订阅
- 日志：日志级别用户可选
- 更新：支持自动检查与静默下载；由用户手动触发检查；安装提示延迟到下次启动
- 上报：仅本地日志与错误记录，不进行上报
- 系统代理：当前用户范围；代理开启时自动切换系统代理
- 配置落盘：App 沙箱路径（最佳实践）
